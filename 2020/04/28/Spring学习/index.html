1.什么是spring

1.1Spring是一个开发应用框架

轻量级、非侵入式、一站式、模块化，

https://spring.io/projects/spring-framework

1.2目的

是用于简化企业级应用程序开发。

Spring是一个超级粘合平台，除了自己提供功能外，还提供粘合其他技术和框架的能力，从而使我们可以更自由的选择到底使用什么技术进行开发

只需要专心你的业务逻辑

常见的「工厂模式」、「代理模式」等，总共有23种模式，利用别人总结的轮子来进行我们自己轮子建造

同时由spring来操作，一定程度上低耦合、高内聚。业务逻辑开发

1.3Spring带来了什么

简化开发

  1. Spring能帮我们根据配置文件创建及组装对象之间的依赖关系。
  2. Spring面向切面编程能帮助我们无耦合的实现日志记录，性能统计，安全控制。
  3. Spring能非常简单的帮我们管理数据库事务。
  4. Spring还提供了与第三方数据访问框架（如Hibernate、JPA）无缝集成，而且自己也提供了一套JDBC访问模板，来方便数据库访问。
  5. Spring还提供与第三方Web（如Struts、JSF）框架无缝集成，而且自己也提供了一套Spring MVC框架，来方便web层搭建。
  6. Spring能方便的与Java EE（如Java Mail、任务调度）整合，与更多技术整合（比如缓存框架）。

1.4底层

1.5小结

Spring核心是IoC容器，要透彻理解什么是IoC容器，如何配置及使用容器，其他所有技术都是基于容器实现的；理解好IoC后，接下来是面向切面编程，首先还是明确概念，基本配置，最后是实现原理，接下来就是数据库事务管理，其实Spring管理事务是通过面向切面编程实现的，所以基础很重要，IoC容器和面向切面编程搞定后，其余都是基于此实现。

2.Spring IoC容器和Bean简介

​ 控制反转（IoC）原理的Spring框架实现。IoC也称为依赖注入（DI）

​ 在Spring中，构成应用程序主干并由Spring IoC容器管理的对象称为bean。Bean是由Spring IoC容器实 例化，组装和以其他方式管理的对象。否则，bean仅仅是应用程序中许多对象之一。Bean及其之间的依赖关系 反映在容器使用的配置元数据中

通常使用IOC容器来管理对象不需要自己再进行new制造对象了

2.1容器概述

ApplicationContextSpring提供了该接口的几种实现。在独立应用程序中，通常创建ClassPathXmlApplicationContext 或的实例 FileSystemXmlApplicationContext。

通常使用xml的时候使用以上两种：

ClassPathXmlApplicationContext：调用类文件的位置一般在此项目中。

FileSystemXmlApplicationContext：调用任意位置的文件。

通常使用spring来管理对象并且从中拿出对象不需要修改源代码且自主new对象出来

8.2 AOP入门

Spring切面可以应用5种类型的通知：

前置通知（Before）：在目标方法被调用之前调用通知功能；
后置通知（After）：在目标方法完成之后调用通知，此时不会关心方法的输出是什么；
返回通知（After-returning）：在目标方法成功执行之后调用通知；
异常通知（After-throwing）：在目标方法抛出异常后调用通知；
环绕通知（Around）：通知包裹了被通知的方法，在被通知的方法调用之前和调用之后执行自定义的行为；

AOP相关了解

Joinpoint(连接点) : 所谓连接点是指那些被拦截到的点。在spring中,这些点指的是方法,因为spring只支持方法类型的连接点.

Pointcut(切入点) : 所谓切入点是指我们要对哪些Joinpoint进行拦截的定义.

Advice(通知/增强) : 所谓通知是指拦截到Joinpoint之后所要做的事情就是通知.通知分为前置通知,后置通知,异常通知,最终通知,环绕通知(切面要完成的功能)

Introduction(引介) : 引介是一种特殊的通知在不修改类代码的前提下, Introduction可以在运行期为类动态地添加一些方法或Field.

Target(目标对象) : 代理的目标对象

Weaving(织入) : 是指把增强应用到目标对象来创建新的代理对象的过程.
spring采用动态代理织入，而AspectJ采用编译期织入和类装在期织入

Proxy（代理） : 一个类被AOP织入增强后，就产生一个结果代理类

Aspect(切面) : 是切入点和通知（引介）的结合

10.事务

10.1四大要素：ACID

一致性：事务结束时，所有的内部数据结构（如 B 树索引或双向链表）都必须是正确的

例如，当开发用于转帐的应用程序时，应避免在转帐过程中任意移动小数点。

原子性：对于其数据修改，要么全都执行，要么全都不执行

持久性：事务完成之后，它对于系统的影响是永久性的。该修改即使出现致命的系统故障也将一直保持。

隔离性:由并发事务所作的修改必须与任何其它并发事务所作的修改隔离。事务查看数据时数据所处的状态，要么是另一并发事务修改它之前的状态，要么是另一事务修改它之后的状态，事务不会查看中间状态的数据。这称为可串行性
